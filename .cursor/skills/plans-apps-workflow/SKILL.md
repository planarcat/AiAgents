# Plans / Apps 工作流

## 目录约定

- **`Plans/`**：存放与用户的**讨论方案**、决议与修订；不经用户要求不在此写代码实现。
- **`Apps/`**：**正式开发**（代码、可运行工程、与实现直接相关的配置）；以已确认的 Plans 为依据推进。

## Plans 内层级

```
Plans/
  <批次文件夹>/
    <主题或序号文件>.md
```

- **批次文件夹**（二选一或按需统一）：
  - **日期+序号**：`YYYYMMDD` + 两位序号，例如 `2026040201`（2026-04-02 当天第 1 个议题批次）。**不同主题线（如「仓库工作流规范」与「具体产品方案」）应使用不同批次目录**，避免流程类记录与产品类记录混在同一文件夹。
  - 或其他与用户约定、在批次内一致的唯一命名。
- **文件**：如 `MVP方案01.md`、`02-鉴权设计.md`；名称需能区分主题，避免整批只有一个模糊文件名。

### 何时新开「方案设计集合」（批次目录）

- **同一轮讨论**的多个切片（`01-…`、`02-…`、综合 **`执行方案.md`**）始终放在**同一批次文件夹**内。
- 当某批次已经产出 **`执行方案.md`** 且本轮讨论进入**新的能力线 / 新议题 / 新史诗**（与「该批次那一轮综合结论」无隶属、仅是产品后续扩展）时：必须在 **`Plans/<新日期+序号>/`** 下**新开一批**，从 `01-…md` 写起。
- 当某批次 **`执行方案` 已按下文第六节「执行方案「已执行」与后续批次」结案**（重命名为 `执行方案（已执行）.md` 或文内标记「已执行」）后：**一律**新开批次承载新设计，**除非**用户明确指定仍修改该旧批次目录。
- **禁止**：仅因「还是同一个 `Apps/` 工程」就把新方案做成旧批次里的 `04-…`，仿佛延续同一轮 Plan；除非用户**明确说**「仍在某某批次目录里补充」。
- **跨批次**：新文件内用相对链接 **引用** 旧批次已定案汇总结论即可——已结案则以 **`执行方案（已执行）.md`** 为准（若仅文内标记而未改名，则仍链接同名 `执行方案.md`）；**不要**为了「好找」去改旧批次已结案文档的关联列表或修订记录（除非用户要求修订该批次已定稿内容）。

## 单个方案文件必须包含

每份 Markdown 建议含以下小节（标题可略调，内容不可缺）：

1. **用户问题**：原意与用户表述要点（可引用原话）。
2. **问题解析**：范围、约束、风险、与已有决议的关系。
3. **解答或实现方案**：可执行结论、备选方案、待决项与依赖。

## 新文件 vs 改旧文件

- **默认**：**新问题 → 新文件**，避免在旧记录上做覆盖式改写导致历史丢失。
- **例外**：问题**简单**且与**上一份记录强相关**时，允许在同文件做**补充或修正**（建议用「修订记录」或「补充（日期）」小节标出变更原因）。

## 综合「执行方案」

当用户明确要求时，在当前**同一批次文件夹**内新增一份汇总文件，例如：

- `执行方案.md` 或 `执行方案-综合.md`

其中应整合本批次（或用户指定的多份）讨论结论，形成**单一、可执行的**步骤清单：范围、里程碑、技术选型决议、接口/数据约定、验收标准；避免仅堆叠片段而缺少优先级与依赖说明。

### 批次边界（勿混为「全仓唯一总方案」）

- 每一份 **`执行方案.md`**（或用户命名为「最终执行方案」的同类汇总）**只对本批次那轮讨论负责**，即「**当前批次下讨论出来的最终方案**」。
- **与其他批次无隶属关系**：不存在跨批次自动生效的「母文档 / 子文档」层级；另一批次的 `执行方案` **不**因文件名相同就成为本批次的延长线。
- **需要时的衔接**：仅通过文内 **相对链接引用** 他批已定结论（例如沿用密钥纪律、技术栈），属 **工程或叙事上的接续**；被引用处若需修订，**只改该批次自己的 Plan**，除非用户明确要求同步修改引用方批次。

## 执行方案落地后的产出与未执行方案累积

### 执行后必须标记「已执行」

当某批次 **`执行方案.md` 已在 `Apps/` 侧按文内里程碑执行完毕**（或用户**明确宣告该案结案**）后，**必须**完成「已执行」标记（见下文 **「执行方案「已执行」与后续批次」**）：重命名为 **`执行方案（已执行）.md`** 或在正文显著位置标注 **已执行**。**禁止**长期保留「已落地却未标注已执行」的 `执行方案.md`。

### 执行后产出（必须）

结案时除「已执行」标记外，**须**在同批次目录内产出下列内容（可合并为一份文件或分两节，仓库内宜统一）：

1. **已执行结果**：简明列出本批次**实际完成**的里程碑、验收结论、关联的 `Apps/` 变更或提交说明要点。
2. **未执行方案**：列出本批次及讨论中**仍未落地**且**仍有效**的项，须覆盖（至少分类说明）：
   - 讨论过但未纳入本次执行的方案；
   - 待执行 / 待设计；
   - 执行失败、待重试或待修复的项。  
   **须排除**：本批次讨论中**明确放弃**或**剔除**的项（在文中注明「已弃用」以免下一批误引用）。

**推荐文件名**：与 `执行方案（已执行）.md` 并列的 **`执行结果与未执行方案.md`**；或全部写在 **`执行方案（已执行）.md`** 文末的「已执行结果」「未执行方案」两节。

### 未执行方案的跨批次累积

- **累积规则**：**本批次文末汇总的「未执行方案」** = **上一批次已累积的全部未执行项**（允许以**引用**上一批 `未执行方案.md` 或 `执行方案（已执行）.md` 中相应小节为主，避免重复粘贴），**减去**本批次执行方案中**已执行**掉的条目，**减去**本批次及讨论中**明确去除**的条目。
- **载体**：每批次目录下推荐维护 **`未执行方案.md`**，内容为「经本批次更新后的**累积**未执行清单」（含指向更早批次的引用链）；若仓库未单建该文件，则须在 **`执行方案（已执行）.md`**（或 `执行结果与未执行方案.md`）中写明**同一套**累积规则下的清单。
- **新开批次的第一版设计稿**：在 **`Plans/<新批次>/` 目录内第一份**设计讨论文件（如 `01-….md` 的**初稿**）**必须**先包含一节 **「自上一批次未执行项」**：以**相对链接**或**摘录**引入**上一批次**的 `未执行方案.md`（或等效小节），再展开本批次新议题。**禁止**在首版设计稿中**从零开始**而完全不承接上一批未执行清单——除非用户**明确声明**整批弃用历史未执行项。
- **去重与剔除**：若某项在本批次**已执行**，从**下一批**累积清单中删除；若某项在讨论中**被用户否决**，从累积清单中删除并可在修订记录中记一笔。

## 执行方案「已执行」与后续批次

当某批次的 **`执行方案.md` 已在 `Apps/` 侧按文内里程碑执行完毕**（或用户**明确宣告该案结案**）后：

1. **先完成上文「执行方案落地后的产出」**（已执行结果 + 未执行方案），再按下列方式标记结案。
2. **后续方案设计默认必须新开批次**：在 **`Plans/<新日期+序号>/`** 下新建文件夹与 `01-…`、`执行方案.md` 等；**禁止**把新一轮议题继续塞进「已结案批次」——**除非**用户**明确说明**要在该旧批次目录内更新（例如：「仍在 `2026040401` 里补充验收修订」）。
3. **标记「已完成执行」**（仓库内任选一种为主、或文件名与文内并用；**同一仓库宜统一**）：
   - **改文件名**：将 `执行方案.md` 重命名为 **`执行方案（已执行）.md`**（或团队约定的等价名称，如 `执行方案-已结案.md`）；Git 历史可追踪重命名。若批次内曾有多处引用「`执行方案.md`」，应 **同步更新相对链接** 指向新文件名。
   - **仅在正文标记**：保持文件名为 `执行方案.md`，在文首（标题下）增加固定状态块，例如：  
     `> **执行状态**：已执行 · 结案日期：YYYY-MM-DD`  
     必要时在 **修订记录** 末行写明「本批次执行完毕，后续议题请开新批次」。
4. **进行中的批次**：未结案前 **不要** 使用「已执行」文件名；可选在文首标注 `> **执行状态**：进行中`。
5. **新批次引用**：新开批次中的 `执行方案` 或 `01-…` 应通过 **相对路径** 引用旧批 **`执行方案（已执行）.md`**（或仍名为 `执行方案.md` 但文内已标已执行的文件），避免复制粘贴旧文导致双源。

## 方案设计 / 讨论设计方案 / 讨论方案（用语触发 · 仅设计）

当用户用类似表述**框定当前回合为设计讨论**时，助手必须按本节执行，直至用户**另行**明确要求落地实现：

**触发用语（示例，非穷举）**：进行方案设计、讨论设计方案、讨论方案、仅讨论方案、先讨论方案、设计阶段、先把方案说清楚、只谈架构不写代码、接下来讨论设计方案，以及同义的英文如 *design discussion only*、*let's discuss the approach first*。

**必须做的事**

1. **默认不改动 `Apps/` 实现代码**：不新增/修改业务逻辑、不为了「演示方案」而直接改 `chat.rs` 等；不主动跑 `cargo` / `pnpm` 作为实现验证，除非用户同时要求实现或验证。
2. **交付物限于设计层**：问题与目标、约束与假设、可选架构或流程、数据/调用关系、利弊与风险、分期与待决项；需要时用**伪代码**、**现有文件路径与职责说明**辅助理解，而非直接提交补丁。
3. **不把「怎么做」默认同「现在就改代码」**：用户问「怎么做」时，先给出**文字方案**（步骤、模块边界、与现有代码的衔接点）；若认为实现有价值，**用一句话询问**是否需要在 `Apps/` 中按该方案修改，**不得**在未获同意时改仓库。
4. **与 `Plans/` 的关系**：若用户希望留档，按上文「目录约定」与「批次」写入或增补 `Plans/<批次>/`；**仅讨论、不落盘**也合法，不强迫写文件。
5. **延续同一话题的子问题**（例如紧接着问某细节的「怎么做」）：**仍视为设计讨论**，除非用户明确切换为「实现 / 落地 / 改 xxx 文件」。

**结束仅设计模式的方式（任一即可）**

- 用户明确说：实现、开发、落地、按方案写代码、改 `某路径`、帮我写进项目、开始写 PR 等；
- 用户说：就按这个方案改、可以动手了、在 Apps 里实现一下。

**与「先方案后开发」的关系**：本节是「**仅方案**」的强化规则；用户一旦结束仅设计模式，再适用上文「先方案后开发」与 `Apps/` 实现流程。

## 助手行为要点

1. **落盘 `Plans/` 前**：若任务涉及「记方案 / 新开主题 / 综合执行方案 / 批次归类」，**必须先重读本 Skill 全文**，再选批次目录；不得仅凭目录里已有 `01`～`03` 就继续在同一文件夹追加 `04`。若目标仓库中**上一批 `执行方案` 已标记为已执行 / 已重命名**，新议题 **默认新开批次**（见上文 **「执行方案「已执行」与后续批次」**）。
2. **新开批次首份设计稿**：在 `Plans/<新批次>/` 下创建**第一版** `01-…`（或等价首文件）时，**须先**按上文 **「未执行方案的跨批次累积」** 引入上一批次的未执行清单（引用或摘录），再写本批新议题。
3. **执行方案结案**：某批次 `执行方案` 在 `Apps/` 落地完毕后，**须**标记 **已执行**，并产出 **已执行结果** + **未执行方案**（累积规则见上文）；不得遗漏。
4. **先方案后开发**：在 `Plans/` 落盘或增补后，再在 `Apps/` 动手实现（除非用户明确只要 Quick patch 且声明不写 Plan）。若用户已用本节「仅设计」用语框定回合，**在对方明确允许实现前**不得动手改 `Apps/`。
5. **写 Plans 时用 UTF-8**，路径与文件名与仓库其余部分保持一致（Windows 下注意非法字符）。
6. **开发活动**默认发生在 `Apps/`；与某 Plan 文件对应时，可在提交信息或 README 中注明关联的 Plan 路径，便于追溯。
