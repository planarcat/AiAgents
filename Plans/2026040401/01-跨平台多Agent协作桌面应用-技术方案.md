# 01 — 跨平台多 Agent 协作桌面应用（技术方案）

## 用户问题

在跨平台桌面端实现一款应用，能力包括：

- 创建若干个 **AI Agent**；每个 Agent 可自定义若干个 **Skills**。
- 用户可与**任意一个** Agent 对话，命令其完成**其能力范围内**的事务。
- **Agent 之间可互相联络**：某一 Agent 可按需向其他 Agent **发起事务请求**；被请求方在**同意**后，利用自身 Skills 获取信息并回复请求方。

**用户举例**：存在「制定旅游攻略」与「介绍指定国家风土人情」两个 Agent。用户要制定攻略时，前者可为某一国家地区生成攻略；当它判断需要风土人情补充时，可向后者发起请求；后者同意后以其 Skills 返回信息，供前者整合。

用户要求：将上述讨论按仓库 **Plans / Apps 工作流**落盘为可追踪方案（本文件 + 同批次 `执行方案.md`）。

## 问题解析

### 范围

- **本期方案**：架构抽象、技术选型建议、Agent/Skill/编排协议、MVP 切片与安全要点。
- **不在本期方案中展开**：具体 UI 视觉稿、商业上架流程、某一云厂商独占对接细节（仅要求 LLM 接口可插拔）。

### 约束

- **跨平台桌面**：Windows / macOS / Linux 至少覆盖主流桌面环境。
- **本地优先**：配置与会话需可持久化；密钥不应在渲染层明文暴露。
- **能力边界**：每个 Agent 仅可调用其绑定的 Skills；跨 Agent 行为需经统一编排层鉴权。

### 风险

- **模型胡编或过度委派**：需 Hub 层超时、白名单、可选人工确认。
- **Skill 执行安全**：任意脚本/子进程必须具备沙箱、超时与资源上限。
- **上下文爆炸**：多 Agent 委派链路过长时需摘要与 `correlation_id` 关联，避免 token 失控。

### 与已有决议的关系

- 本批次为 **新开议题**（仓库内原无 `Plans/` 记录），不与旧批次方案冲突。

## 解答或实现方案

### 1. 目标抽象

系统本质是：**多 Agent 运行时 + 可插拔 Skills + 带策略的 Agent 间 RPC（经 Hub）**。

桌面端职责：配置与持久化、对话 UI、编排与安全边界。模型与工具层职责：是否委派、如何填参数、如何归纳回复。

### 2. 技术选型（跨平台桌面）


| 方向                                        | 说明                                                                      |
| ----------------------------------------- | ----------------------------------------------------------------------- |
| **优先：Tauri 2 + 前端 React**                   | 已定 **React + shadcn/ui + Tailwind**；Rust 承载编排与安全关键路径。                    |
| **备选：Electron + Node**                    | 生态成熟、全栈 TypeScript 友好；安装包通常更大。                                          |
| **编排核心**                                  | **已定：Rust（Hub）+ TS（UI）**；消息路由、持久化、MCP 与策略见 **[执行方案.md](./执行方案.md) §2.1**。   |


**LLM**：**OpenAI 兼容 HTTP API**（云端或自建网关）；**首版与当前决议周期内不考虑本地模型**（不捆绑 Ollama / 本地权重推理），以降低安装包与设备矩阵复杂度。

**Skills 与 MCP**：外部能力以 **MCP（Model Context Protocol）** 为**标准接入方式之一**（与 builtin 并列）；Agent 绑定的 Skill 可直接映射到已配置的 MCP 工具（含 `stdio` / **SSE** 等传输，具体以 `Apps/` 实现为准），由 Hub 统一拉起、鉴权与超时。

**持久化**：**SQLite**（Agent 配置、会话、委派记录、Skill 结果摘要）。

### 3. 核心架构（四层）

1. **Agent 定义层**
  - 字段建议：`id`、系统提示、绑定的 Skills 列表、可呼叫的 **其他 Agent 白名单**、接受委派策略（自动 / 模型判定 / 用户确认）。
2. **Skill 层（Tool）**
  - 统一抽象：`name`、**给模型用的描述**、JSON Schema 参数；实现类型包括 **builtin**、**MCP**（推荐用于对接外部服务与数据源）、HTTP、（可选）沙箱脚本。  
  - **MCP Skill**：在配置中声明 MCP Server 连接方式与暴露的工具子集；运行时由 Hub 完成 `list_tools` / `call_tool` 并与 LLM 的 function calling 对齐。  
  - 每个 Agent 仅挂载允许的 Tool 子集。
3. **编排 / Hub**
  - 所有 A→B 请求经 Hub：**鉴权、配额、审计、超时、重试**。  
  - 对模型暴露结构化动作，例如：`delegate_task(target_agent_id, goal, context_ref, deadline_opt)`。
4. **对话与上下文层**
  - 每 Agent 独立会话；委派在 Hub 侧产生 **子任务**，带 `correlation_id` 挂回主会话；支持异步 Job + SQLite 状态机。

### 4. Agent 间协议（最小消息模型）

- **Request**：`from`、`to`、`intent`（自然语言或结构化 goal）、`payload`（结构化 + 引用 ID）、`reply_to`（关联主会话）。  
- **Response**：`accepted` | `rejected` + `reason`；若接受，再异步 **Result**（可含多段 partial）。

**「同意」的实现路径（由浅入深）**：

- **MVP**：被叫 Agent 用自身系统提示 + LLM 做一次结构化「是否接受 / 能力是否匹配」判断后再执行。  
- **加强**：高敏 Skill 或跨 Agent 调用强制 **用户二次确认**。  
- **可选规则**：关键词 / intent 类型白名单，减少无效模型调用。

### 5. 安全与沙箱

- 可执行 Skill：**子进程隔离、超时、CPU/内存上限**；禁止对用户输入直接 `eval`。  
- **Agent 图**：Hub 校验「A 是否允许呼叫 B」；全量委派日志便于审计与排错。  
- **密钥**：API Key 走系统凭据或 OS Keychain，不经由前端明文传递。

### 6. MVP 切片（建议迭代顺序）

1. 单用户、SQLite、2～3 个预置 Agent；Skills 含 **至少一条 MCP 路径**（可与 builtin 并存）。
2. Hub 实现同步 `delegate`：请求—同意—执行—单段结果回传。
3. UI：Agent 切换、会话列表、委派链路可视化（pending / done）。
4. 扩展：异步子任务、可视化配置 Agent/MCP Server、更细粒度的同意策略。

### 7. 已定稿技术栈摘要

| 项目 | 决议 |
|------|------|
| UI | **React** + **shadcn/ui** + **Tailwind CSS**（构建 **Vite**；详见 **[执行方案.md](./执行方案.md) §2.1**） |
| Hub | **Rust**（Tauri 后端：持久化、LLM、MCP、委派核心）+ **TypeScript**（仅前端 UI 与 `invoke`） |
