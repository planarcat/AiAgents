## 1. 范围（本执行方案覆盖）

- 跨平台桌面壳（优先 Tauri 2 路线）。
- 多 Agent 配置、Skills 注册与绑定（**含 MCP**，与 builtin 同属首批能力）。
- 用户与单 Agent 对话；Hub 支撑的 **Agent 间委派**（含同意/拒绝与结果回传）。
- 本地 SQLite 持久化、基础安全（白名单、委派日志、Skill 超时）。

**明确不包含（首版）**：云端多用户同步、复杂 P2P Agent 发现、商城化 Skill 分发。

## 2. 技术选型决议




| 决策项          | 决议                                   | 备注                                                                    |
| ------------ | ------------------------------------ | --------------------------------------------------------------------- |
| 桌面框架         | **Tauri 2**（首选）                      | 若团队坚持全 JS，可备选举 Electron，需在 Apps README 中记录偏离原因                        |
| 持久化          | **SQLite**                           | 会话、Agent 定义、委派记录、任务状态                                                 |
| LLM 接入       | **OpenAI 兼容 API（远程）**                | 配置 base URL / Key；**首版不考虑本地模型**；**对话窗会话级模型预设**（首期 **DeepSeek** 可选）见 **[2026040402/02](../2026040402/02-对话窗口与模型选择.md)**                                       |
| Skills / MCP | **MCP 为一等接入**                        | Agent 可绑定 MCP 暴露的工具；Hub 负责连接生命周期与安全策略（超时、允许列表）                        |
| 编排           | **集中式 Hub**                          | 所有跨 Agent 消息经 Hub，禁止渲染层直接互调                                           |
| 前端 UI        | **React + shadcn/ui + Tailwind CSS** | 构建见 **§2.1**；组件基座 **Radix**（shadcn 默认）                                |
| Hub 实现       | **Rust + TypeScript**                | **Rust** 承载持久化、LLM 代理、MCP、委派状态机与密钥；**TS** 仅 UI 与 `invoke`，不持有 API Key |

- **React** → 仅展示状态与用户操作，通过 **Tauri command** 调用 Hub。

## 3. 接口与数据约定（MVP）

### 3.1 Agent 配置（逻辑模型）

- `id: string`
- `display_name: string`
- `system_prompt: string`
- `skill_ids: string[]`
- `allows_outgoing_delegation: boolean`（**如需其他助手辅助时是否允许向外委派**：当本助手认为事务自身无法完成时，是否允许委派给其他助手；见子批次 [2026040402 / 01 · §5.1](../2026040402/01-默认Agent与内置委派Skill.md)；默认建议 **true**；为 **false** 时 Hub **禁止**该 Agent **发起** `DelegateRequest`）
- `accepts_incoming_delegation: boolean`（**是否允许被其他助手委派**：见子批次 [2026040402 / 01 · §5.2](../2026040402/01-默认Agent与内置委派Skill.md)；默认建议 **true**；为 **false** 时 Hub **拒收**入站 `DelegateRequest` 且该 Agent **不列入**委派目标发现/枚举）
- `accept_policy: "llm" | "auto" | "user_confirm"`（可选扩展；在 **已允许入站** 的前提下约束被叫侧如何接受/拒绝）
- `delegate_allowlist: string[]`（**可选扩展，MVP 不作为门禁**：**首版**委派是否允许由 **`allows_outgoing_delegation`** 与目标 **`accepts_incoming_delegation`**（及 §3.5 配额等）共同判定，**无需**逐目标白名单；本字段预留供后续「仅允许委派给部分 `agent_id`」类产品选项；**实现可忽略**直至该选项落地）

### 3.2 Skill（Tool）注册

- `id`, `name`, `description`（模型可见）, `parameters: JSON Schema`, `handler` 类型（**builtin | mcp** | http | sandbox）
- **MCP 类**：额外字段建议包含 `mcp_server_id`（或内嵌连接配置）、`tool_name`（MCP 侧工具名）；具体 schema 在 `Apps/` 配置模型中定稿。

### 3.3 委派消息（Hub）

- **DelegateRequest**：`request_id`, `from_agent_id`, `to_agent_id`, `intent`, `payload`（JSON）, `parent_message_id` / `correlation_id`
- **DelegateResponse**：`request_id`, `status: accepted | rejected`, `reason?`
- **DelegateResult**：`request_id`, `content`（文本或结构化摘要）, `partial?: boolean`

### 3.4 数据库表（建议最小集）

- `agents`, `skills`, `agent_skills`, `conversations`, `messages`, `delegations`, `jobs`（异步任务时用）
- **`conversations` 与会话级 LLM 路由**：可增 **`llm_preset_id`**（或等价）以支持 **对话窗口可选大模型**，细则见 **[批次 2026040402 / 02](../2026040402/02-对话窗口与模型选择.md)**。

（具体列类型在 `Apps/` 迁移脚本中细化。）

### 3.5 Hub 配额与限流（MVP **默认值**，可配置）


| 参数                          | 建议默认   | 含义                                                                          |
| --------------------------- | ------ | --------------------------------------------------------------------------- |
| **委派链最大深度**                 | **3**  | 仅统计 **Agent→Agent** 边：从当前会话根到本次委派目标的路径长度；超过则 Hub **拒绝**并返回可观测错误（防上下文与成本失控）。 |
| **单会话（conversation）最大委派次数** | **32** | 同一会话生命周期内累计 `DelegateRequest` 成功受理次数上限（含嵌套委派）；超出拒绝。                         |
| **单次模型回合内最大委派次数**           | **8**  | 从一次用户发起到助手本轮回复结束之间，允许的委派次数上限；防止单轮扇出爆炸。                                      |


**说明**：上述数值面向首版桌面单机场景；若后续支持超长自动化任务，可改为「按订阅档位」或按 Agent 单独配额。

**入站委派「积极回应」与超时**：当 `accepts_incoming_delegation` 为真时，被叫须在 Hub 规定的 **`T_accept`** 内给出 **`DelegateResponse`（accepted/rejected）**；**超时**视为**不积极**，Hub 对发起方返回 **rejected**（可观测 timeout 原因）。细则与默认秒数见 **[2026040402 / 01 · §5.2](../2026040402/01-默认Agent与内置委派Skill.md)**。**发起**委派还须主叫 **`allows_outgoing_delegation`**（见该文 **§5.1**）。

## 5. 验收标准（首版）

1. 可在三个平台（或 CI 矩阵至少覆盖 Windows + 其一）**成功构建并启动**。
2. **首装预置的一条 Agent** 存在且可对话（细则与初态 Skill 绑定见 **[2026040402](../2026040402/执行方案.md)**，**不**特殊化）；另有 **至少 1 个用户另行创建或导入的 Agent** 用于验证差异化人设；**每 Agent ≥1 Skill**（首装行初态见 2026040402；其他助手在 M3 至少绑 1 条 MCP 或 builtin）；全应用至少存在 **一条 MCP Skill 调用成功路径**；助手切换无串会话。
3. **委派闭环**：在 A **`allows_outgoing_delegation`**、B **`accepts_incoming_delegation`** 均为真（且满足 Hub 配额、深度等）时，A 可向 B 发起委派；B 拒绝或接受均有明确状态与日志；接受后 B 的 Skill 结果能回到 A 侧并体现在最终回复或可追溯记录中。若 A 关闭出站或 B 关闭入站，Hub **明确拒绝**对应路径。**MVP 不以 `delegate_allowlist` 为必要前置条件**（见 §3.1）。
4. 应用重启后会话与配置**可恢复**（或文档声明哪些仅存内存，需一致）。
5. 无明文将 API Key 写入前端 bundle；配置中的密钥读取路径可追溯。
6. **MCP 首版 UX/文档**：帮助或设置中可见「**本地优先 stdio，远端 SSE（HTTPS）**」；新增/首次连接 MCP 时走**确认**流程，且 `**stdio` 展示完整启动命令**，**SSE 拒绝非 HTTPS**；组织级域名列表行为与 §6.3 一致。

## 6. Hub 限流与 MCP 策略（本批次已定稿）

> 下列 **transport / Server 策略**经用户确认：**按文中决议实施，不再作为开放选项摇摆**。

### 6.1 委派限流（与 §3.5 一致）

已采纳默认值见 **§3.5**（深度 **3**、每会话 **32** 次、每用户轮次内 **8** 次）。实现时所有阈值应**集中可配**（如配置文件或设置页高级项），并在 Telemetry/日志中打满计数器便于调参。

### 6.2 MCP 传输：`stdio` vs **SSE**（对比参考 + **已定稿决议**）


| 传输                   | **优势**                                         | **劣势**                                                        | **首版定位**                  |
| -------------------- | ---------------------------------------------- | ------------------------------------------------------------- | ------------------------- |
| **stdio**            | 本地子进程，链路简单；无额外 HTTP 攻击面；易与 CLI 类 MCP Server 集成 | 需 Hub 负责**拉起/守护进程**；Windows 下命令行/路径/引号易踩坑；难直连「仅在远端暴露」的 Server | **默认主推**：本机、开发期、可信包内的 MCP |
| **SSE**（及同类 HTTP 流式） | 可连**远程** Server；与网关、多租户托管形态兼容                  | 依赖网络与 TLS；URL/Token 泄露风险；需严格**HTTPS、令牌轮换与域名允许列表**             | **并行支持**：远端 MCP、团队统一托管    |


**首版_transport 决议（已定稿）**：`**stdio` 与 SSE 均实现**；**UI 与产品文档**须明示：「**本地优先 `stdio`**，**远端用 SSE（HTTPS）**」。

### 6.3 允许连接的 MCP Server：策略对比 + **已定稿决议**


| 策略                                                 | **优势**                           | **劣势**                                | **首版建议**                                 |
| -------------------------------------------------- | -------------------------------- | ------------------------------------- | ---------------------------------------- |
| **仅用户配置**（手动添加 Server，配置存本机）                       | 实现快、兼容所有来源；权力边界清晰（用户自负）          | 社会工程 / 误配可导致**恶意进程**或**外送数据**；无「官方背书」 | **首版基线**（见下行决议合并实施）                      |
| **首次连接确认**                                         | 在「仅配置」之上增加一次显式同意，可展示将要执行的命令或 URL | 多 Server 时易弹窗疲劳；仍不防「用户自己点同意」          | **首版强制**（与显式用户配置组合，见下行决议）                |
| **命令/命令行模板 allowlist**（例：仅允许 `npx @scope/xxx` 某前缀） | 显著降低任意二进制执行面；适合企业环境              | 维护成本高；开源生态需要持续更新白名单                   | **企业版或高级安全模式**再上                         |
| **签名 / 哈希 / 目录.catalog**                           | 强保证二进制与发布源一致；可做自动更新与审计           | 需要签名基础设施、镜像与法务流程；对小团队重                | **后续里程碑**；可与「精选 MCP 目录」一起做               |
| **远端 SSE 固定域名白名单**                                 | 限制数据外发到未知主机                      | 与 SaaS 灵活性冲突；需配套代理                    | **首版可选**：独立配置文件，**默认关闭**（关闭时不校验域名；见下行决议） |


**首版_Server 策略决议（已定稿）**：**显式用户配置 + 首次连接确认**；`**stdio`** 在确认流程中必须展示将要启动的**可执行路径与完整参数**；**SSE** **仅允许 HTTPS**（非 HTTPS 端点拒绝连接）；提供可选「**组织级域名允许列表**」配置文件——**默认关闭**，关闭时**不**做域名校验，开启则仅允许列表内主机。

**已决议（本批次既有）**：首版 **不纳入本地模型**；**Skills 即可接 MCP**，列入 M3 验收与大纲范围。
