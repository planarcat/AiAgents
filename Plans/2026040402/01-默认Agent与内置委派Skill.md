# 01 — 初装预置助手与内置 Skill 种子

## 用户问题

应用在安装完成、用户初次打开并配置 **API Key** 后，列表里 **已有一条可对话的助手**，无需先手动新建。该条 **不是**「特殊身份」的 Agent：只是 **开发者在首装时替用户执行了恰好一次** 的创建操作，且与用户在 **新建助手** 流程里点击 **「默认模板」** 所使用的逻辑 **完全相同**——得到的是 **仅绑定「必要 Skills」、不绑任何其它多余 Skill**（无 MCP、无额外 builtin）的一条普通 `agents` 记录。**必要 Skills** 现时至少含 **委派**、**规则类**（如禁编造）；二者属于 **一般情况下所有 Agent 都应具备** 的基线；**未来** 若产品定义更多 **全局必须或建议安装** 的 Skill，也纳入同一 **必要 Skills 列表**，**首装生成** 与 **「默认模板」** 均只应用这份列表，**开发者不对首装那条记录做任何与其它 Agent 不同的特殊处理**。

## 问题解析

### 范围

- 约定：**全局 `skills` 定义**、**「必要 Skills」清单（可版本迭代）**、**首装一次性创建**（与 **默认模板** 同路径）、以及 **新建助手时的「默认模板」按钮**。
- **与父批次** **[执行方案（已执行）](../2026040401/执行方案（已执行）.md)**：Hub、委派 DTO、配额、MCP 已定稿策略不变。

### 约束（不对预置行特殊化）

- **禁止** 出现仅对「默认 / 种子 / 固定 UUID」生效的业务分支：例如 **禁止** `if agent_id == SEED` 绕过委派 **出站/入站策略**、配额或审计；**禁止**「只有预置助手才能绑委派 builtin」等硬编码。
- **委派能力** 仅来自 **是否绑定了委派类 builtin**，与「是否首装那条」无关。
- **删除**：对预置助手调用 **与删除任意 Agent 相同** 的接口与后果；**禁止** UI 层阻止删除后伪装成「仍有一条隐藏助手」。

### 风险

- 用户删光助手：**新建** 流程中 **「默认模板」** 与其它入口 **同级展示**；**不对「默认模板」做高亮、脉冲、空列表专项引导**（已决）。**禁止**「仅服务首装 UUID」的专属恢复逻辑或文案。
- **Skill 表未落地前**：需先迁移 `skills`、`agent_skills`。

### 与已有决议的关系

- 委派 builtin 在 **M4** 与 Hub `delegate` 对齐；规则类 builtin **MVP 为纯提示档**（见 **§3**、**§7** 已决项）。

## 解答或实现方案

### 1. 概念定义（仅产品叙事，非运行时类型）


| 术语                  | 含义                                                                                                                                                                                                                                                                     |
| ------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **必要 Skills**       | **全局清单**（可由产品/版本迭代）：现时至少含 **委派**、**规则类** builtin；**未来** 扩充「每个 Agent **必须** 或 **建议** 安装」的项时，**首装** 与 **默认模板** 创建出的 Agent **只绑定该清单内 Skill**，**不**自动附加 MCP 或其它非清单 Skill。清单的权威来源建议在 `Apps/` 单列配置或迁移种子中维护，**不作**首装专属分支。                                                     |
| **默认模板**            | **新建助手** 时 UI 上的 **「默认模板」** 能力：一次点击即 **创建一条普通 Agent**，仅绑定 **必要 Skills** + 推荐的 `display_name` / `system_prompt` 初值；**无**其它 Skill。后端即 **与普通 `create_agent` 相同** 的一条代码路径（例如 `use_default_template: true` 或专用 command 内部仍调用同一核心）。**不对该按钮做高亮、引导遮罩、空列表特判强调**（与普通「新建」入口一致对待）。 |
| **首装一次性创建**         | **开发者替用户仅做一次**的事：在 **空库首次**（或约定好的首次启动条件）调用 **与「默认模板」完全相同** 的创建逻辑，插入 **一条** `agents` 行。**之后**对该行 **不作**任何与「是否首装」相关的特殊处理——与普通用户创建的 Agent **完全一致**。                                                                                                                       |
| **委派 builtin**      | `handler=builtin`，`builtin_code=delegate_to_agent`；tool 名例：`request_agent_help`；schema 对齐父批次 **§3.3**。无合适被叫、**主叫出站关闭**、**被叫入站关闭**或 Hub 配额/深度等不允许时 **明确失败**，**不得**编造成功委派。                                                                                                                    |
| **规则类 builtin（首期）** | 例：`honesty_acknowledge`：强化「不能完成则不捏造」；与系统提示互补。                                                                                                                                                                                                                          |


### 2. 首装插入与「默认模板」

- **首装（已决 · 方案 B）**：**迁移** 只负责 **schema** 与 **`skills` / `agent_skills` 等全局种子**（若有），**不在迁移中插入首条 `agents` 行**，**不**为预置助手预留迁移层固定 UUID。应用侧：当 **`app_meta.first_default_template_applied`**（或等价键名，在 `Apps/` 锁版）为 **未执行**，且满足「**无 Agent / 空库**」等产品条件时，**内部调用与「默认模板」完全相同** 的 **`create_agent`** 核心，插入 **一条** 助手（**`agent_id` 当场生成**，与用户在新建里点默认模板一致）；随后 **将 meta 标记为已执行**。**幂等**仅靠 **该 meta 只生效一次**（与迁移重复执行无关）。**业务层不读任何首装种子 UUID**。**不得**在 meta **已为已执行**、用户 **删光 `agents`** 后，仅凭 **重跑迁移** 或 **再次启动** **擅自再插**预置行；用户需 **新建 → 默认模板** 得到同款配置（新 id）。
- **新建助手 · 「默认模板」按钮**：用户在 **创建 Agent** 流程中点击后，生成 **新 `agent_id`**，**仅** 绑定 **必要 Skills**（当前清单），**不**预绑 MCP。与首装那次调用 **共享实现**，仅 **调用次数与触发入口**不同。
- **用户删除首装那条**：与普通删除相同；若需再有一条「同款」——走 **新建 → 默认模板**，得到 **新 id** 的等价配置，**无**「恢复首装 UUID」语义。

### 3. Skill 与绑定（必要 Skills）

**`skills`** 表插入全局 builtin 定义（稳定 `id`，多 Agent 可引用）：

| 建议 `builtin_code` | 作用（摘要） |
|---------------------|-------------|
| `delegate_to_agent` | 委派；无目标/不允许时明确错误 |
| `honesty_acknowledge`（或等价） | 无法完成时声明限制、禁编造 |

**必要 Skills** = 上表所列（及未来迁移中追加的 **全局必选/建议** 项）。**首装创建** 与 **「默认模板」** 均为目标 Agent **仅** 绑定此集合；用户之后可自行 **增绑 MCP 等**；与任意 Agent 相同。

**规则类 builtin（MVP 已决 · 纯提示）**：以 **tool 名称 / description / Schema** 与 **`system_prompt`** 引导模型在无法完成时调用；**handler** 用 **轻量**实现（如固定返回模板、写入本轮上下文字段），**不**对模型最终回复做语义级「是否捏造」判定，**不**为规则类 tool **单开**与委派同级的大型审计表。**「带审计」**（结构化留痕、更强合规策略）留 **后续迭代**，非首版必做。

**委派能否发生**（MVP）：由 **§5** 两处开关与父批 **§3.5** 配额等共同判定；**不**以 **`delegate_allowlist`** 为默认门禁（见 **§5** 首段、**§7** 已决项）。

### 4. 统一规则表（不再二元对比「预置 vs 自建」）

所有 `agents` 行：


| 规则                                           |
| -------------------------------------------- |
| 存储于同一表，同一套编辑 / 删除 / 对话 / 委派 API              |
| Skills 仅由 `agent_skills` 决定；无任何「内置尊号」判据      |
| Hub 配额、**出站/入站委派字段**、审计 **只认 `agent_id` 与绑定**，不认「是否首装」 |


### 5. 编辑助手：出站与入站委派开关

**Hub（MVP）**：是否允许形成一次 **`DelegateRequest` → …**，**不**依赖 **`delegate_allowlist` 逐目标白名单**（父批 §3.1 将该字段标为**可选扩展**，首版可**不读**）。**共同必要**条件包括（另叠加父批 **§3.5** 配额、链深度、主叫是否已绑 **委派 builtin** 等）：

1. **主叫**：**`allows_outgoing_delegation === true`**。否则 Hub **禁止发起**，返回可观测错误（工具/handler/Hub 前置一致）；**不得**假装已委派。
2. **被叫（`to_agent_id`）**：**`accepts_incoming_delegation === true`**。否则 **前置拒绝**（不进入接受流），且不列入「可委派目标 / 发现」。

若将来产品启用「仅允许委派给部分 Agent」选项，再让 **`delegate_allowlist`**（或非空子集约束）参与判题；与当前 **「一般不用白名单」** 约定不冲突。

---

#### 5.1 如需其他助手辅助时是否允许委派（出站）

在 **助手编辑 / 设置** 中增加开关（父批 [执行方案（已执行）§3.1 · `allows_outgoing_delegation`](../2026040401/执行方案（已执行）.md)）：

| UI 文案（建议） | **如需其他助手辅助时是否允许委派** |
| --------------- | ---------------------------------- |
| 控件类型 | **开关**（`boolean`） |
| **功能解释（展示给用户）** | 若本助手 **认为某项事务自己无法完成**，**是否允许委派给其他助手**代为完成。若 **关闭**：即使模型侧暴露委派 tool，Hub 也会 **拒绝**本助手 **向外**发起的委派；用户可配合 **`system_prompt`** 强化「勿在关闭时仍提议委派」。 |
| **默认** | 建议 **开启**；仅希望助手永远自行消化、绝不外派者可关。 |

---

#### 5.2 是否可以被其他助手委派事务（入站）

在 **助手编辑 / 设置** 中增加开关（父批 [执行方案（已执行）§3.1 · `accepts_incoming_delegation`](../2026040401/执行方案（已执行）.md)）：

| UI 文案（建议） | **是否可以被其他助手委派事务** |
| --------------- | ------------------------------ |
| 控件类型 | **开关**（`boolean`） |
| **功能解释（展示给用户）** | 若 **开启**：当其他 Agent **需要协助**并选中本助手时，本助手 **允许**进入 Hub 委派流；应在 **`T_accept`** 时限内给出 **接受或拒绝**等 **可观测终态**（见下）。若 **关闭**：**Hub 对入站 `DelegateRequest`（`to_agent_id` 为本助手）前置拒绝**；且在可委派目标发现中 **不出现**。 |

**Hub 超时与最低限度响应（与「积极」一致，对用户可不逐字展示）**：

- 自 **`DelegateRequest` 到达被叫侧（进入接受判定流程）** 起，Hub 启动 **`T_accept`**（可配置，默认值在 `Apps/` 实现阶段锁版）：被叫须在 **`T_accept` 内** 产生 **`DelegateResponse`（`accepted` 或 `rejected`）** 或 Hub 认可的等价终态。
- **`T_accept` 届满仍未收到明确接受/拒绝** → 视为 **不积极**：Hub **代为结束**该轮，对发起方返回 **`rejected`**（建议 `reason` / 错误码如 **`timeout` / `no_timely_response`**），并打审计日志；**不得**无限 `pending`。
- **执行阶段**（已 `accepted` 之后至 `DelegateResult`）可另设 **`T_execute`** 或沿用既有 Skill/子任务超时（父批 **M6** 等）；**「积极」在回应段主要指 `T_accept` 内的最低限度响应**。数值与是否分列多档由实现定稿。

**默认**：建议 **开启**（便于协作）；用户可为专注私密、不接任务的助手关闭。

**说明**：**§5.1** 与 **§5.2** **相互独立**；二者与 **`accept_policy`** 串联时：**仅在被叫已允许入站（§5.2 开）** 后，才进入 `accept_policy` 所描述的接受判定路径。
