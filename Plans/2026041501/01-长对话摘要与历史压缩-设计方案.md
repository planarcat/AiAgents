## 1. 用户问题

在长对话中，**全量**将 `messages` 拼进每轮 `chat.completions` 会导致：

1. **Token 与成本**逼近模型上限，延迟上升；
2. **注意力稀释**：早期助手表述权重大，与用户当前意图、**当前已装载工具**更容易错位（与 `05` 文档现象同源，摘要是额外杠杆）。

期望：**在必要时压缩历史对话**：保留近期轮次可读性，对更早内容用**可控长度的摘要**替代或折叠，且与现有 **Tools 注入、SQLite 历史真相**一致。

---

## 2. 问题解析

### 2.1 约束

| 维度 | 说明 |
|------|------|
| **真相源** | `messages` 表保留**完整原文**（含 `system`/tool 策略另议见 §3.4）；压缩仅影响**送往模型的视图**，而非默认删库。 |
| **能力权威** | 每轮仍 **`load_tools` + `build_capabilities_system_suffix` + 完整 `tools` 参数**；摘要不得写死「永久无某能力」。 |
| **实现栈** | 继续 **Rust（Tauri Hub）+ async-openai**；摘要生成 = **额外一次 LLM 调用**（可与对话同 API、同 key；模型以当前 DeepSeek 预设为主，未来可配置「摘要用轻模型」）。 |
| **可观测** | 开发模式下可沿用 env/文件日志思想，记录「是否触发摘要、摘要前后 token 估计」便于调试（不与用户必需 UI 绑定）。 |

### 2.2 设计目标（可验收）

- **G1**：当估算上下文超过阈值时，能在**单次用户发送前**完成折叠，使当轮请求落在预算内（或明确降级策略）。
- **G2**：**最近 N 轮** user/assistant **原文**保留（N 可配置，与 1201 §7.9 建议 6～12 轮一致的量级）。
- **G3**：冷区以 **一条滚动会话摘要** 表达（可递归压缩），有**硬字数/token 上限**；
- **G4**：与 **Skills 状态**叙述一致：摘要模板含固定护栏句（§4.3）。

## 3. 方案总览（对齐 1201 §7.9）

采用 **热区 verbatim + 冷区滚动摘要 +（后续）tool 段折叠**。**主触发** = token（或字符 proxy）预算；**事件触发** = 任一 Skill **装载状态**变更（合并为会话级 **`compression_pending`**，详见 §3.5.1）；**辅触发** = 用户显式「总结本会话」等（产品化可后做）。

### 3.1 拼消息顺序（建议）

送入 `CreateChatCompletionRequest` 的 `messages` 逻辑顺序建议为（与现有代码对齐时再微调）：

1. **System**：`agents.system_prompt` + 内置后缀 + Skill L2 + **能力清单后缀**（已有）。
2. **可选 · 会话压缩块**（新）：**单条** `role=system` 或固定标签的 `user`，例如 `【会话摘要（冷区压缩）】…`，放在 **历史 user/assistant 列表之前**、紧跟主 system 之后，避免夹在 multi-tool 中间。**须约定与现有 `TOOLS_CHANGED_NOTICE` 等 system 行的相对顺序**（建议：主 system → 会话摘要 → 指纹类 system 历史行仍按 `created_at` 从 DB 读来？—— **实现时二选一**：要么摘要不入库仅运行时注入；要么入库为特殊 `message_type`/`role`——见 §3.3）。
3. **历史消息**：按时间顺序拼接 **折叠后的视图**（冷区已被摘要替代；热区原文）。

> **原则**：模型永远先看到「能力与安全边界」，再看到「冷区摘要」，再看到「近期原文」。

### 3.2 热区（verbatim）

- 从**最新**向前数，保留最近 **N 个完整对话轮**（建议以「user 消息条数」或「user+assistant 对数」计量，二者取一并在配置中写清）。
- **N** 默认值建议 **8 轮 user**（可调）；或改为 **token 硬顶** 专用于热区，先触顶者为准（与 7.9 一致）。

### 3.3 冷区（滚动摘要）

- **输入**：上一轮已存在的「冷区摘要」文本 + **本次从热区外新移出的**消息片段（或分段摘要再合并）。
- **输出**：**一条**新摘要，长度上限例如 **600～1000 汉字** 或等效 token（配置化）。
- **递归**：`S_{k+1} = Compress(S_k + 新移出块)`，始终只保留 **一份**当前摘要字符串与会话关联存储（见 §5.1）。

### 3.4 Tool / assistant 多轮

当前 `messages` 表主要为 user/assistant；**若**未来或已有 tool 消息在 API 层的表示：**最近 M 次** tool 相关往返尽量保留原文；更早的折叠为「工具名 + 要点」（1201 §7.4）。**MVP 可仅对 user/assistant 做冷热分区**，tool 仍全量或按简单截断 —— 在 **`执行方案.md`** 中标为 **Phase 1.5**。

### 3.5 触发时机

| 触发 | 说明 |
|------|------|
| **主** | 估算 `system + 历史 + 本轮用户` 将超 `budget_max * ratio`（建议 ratio **0.72～0.80**）时，在 `chat_send` 内**先**执行一次摘要更新（或先做热区滑动再摘要）。 |
| **辅** | 用户点击「生成/刷新会话摘要」、或运维开关强制压缩。 |
| **事件（Skill 装载）** | **任一** Agent 的 Skill **装载状态**发生变化（对 `agent_skills` 的增删改，使**绑定集合相对保存前**有差异）时，对该 Agent 下**全部会话**要求：**下一次**构建模型上下文前执行**至多一次**冷区压缩（与现有 [`insert_tools_changed_notices_for_agent`](../../Apps/agent-hub/src-tauri/src/hub/chat.rs) 路径对齐，见 §3.5.1）。 |
| **与变更提示关系** | Skills 变更仍会插入 `TOOLS_CHANGED_NOTICE`（`role=system`）；摘要链路解决「长历史叙事」，与 system 提示**叠加**；摘要模板仍须遵守 §4.3，不以摘要替代当轮能力清单。 |

#### 3.5.1 Skill 变更触发的压缩：**不重复触发**

**目标**：同一轮「保存助手配置」里无论改多条 Skill、或用户**连续多次保存**之间未发新消息，对**同一会话**不得对**同一次用户发送**重复跑两遍压缩 LLM；多次变更在时间上紧邻时，合并为**一次**待处理信号。

**建议实现（会话级 pending + 合并）**

1. 在 **`conversations`** 表增加布尔（或整型 epoch）列，例如：  
   - `compression_pending BOOLEAN NOT NULL DEFAULT 0`  
   含义：本会话**尚有一次**因 Skill 装载变化而需要的上下文压缩；在**下一次** `chat_send` 完成该次压缩逻辑后清零。
2. 在现有 **`agents_update`（或唯一写 `agent_skills` 的保存路径）** 中，当检测到绑定集合相对保存前发生变化、并将调用 `insert_tools_changed_notices_for_agent` **同时**：对该 agent 下**所有** `conversation_id` 执行  
   `UPDATE conversations SET compression_pending = 1 WHERE agent_id = ?`。  
   单次事务内多次改绑定 → **只执行一次**批量 UPDATE，天然合并为**一次未决压缩**。
3. **`chat_send` 内**（在估算 token、拼 `api_messages` 之前）：  
   - 若 `compression_pending == 1` **且** 本会话需要走冷区摘要路径（已有历史可压或超过热区）：执行**本轮唯一一次**压缩更新（更新 `rolling_summary` 等），然后将 `compression_pending = 0`。  
   - 若当前历史很短、仅热区就够、尚不需要调用摘要 LLM：可将 pending **仅用于**强制做一次「轻量折叠」（例如仅热区滑动 + 在摘要块注入一句「能力集刚变更，旧轮次结论请以当轮清单为准」）**或**在实现上将 pending **延迟**到首次满足「需冷区」时再消费 —— **须在实现时二选一写清**，避免 pending 永不清。  
   **推荐**：pending 置位后，**下一轮用户发送**时若仍不需要 LLM 摘要，也**清零 pending**（并可选写一条调试日志），避免无限重复；若需要 LLM 摘要，则一次调用后清零。
4. **与「主触发（token）」同时满足**：同一轮 `chat_send` 只发起 **一次**摘要 LLM（合并理由：`compression_pending` + 超预算），**禁止**因两个条件并列而调用两次。

**与「不重复」的边界**：  
- **不同**的 `chat_send` 轮次：用户发第一条消息后压缩并清零；再改 Skill → pending 再置 1 → 用户发第二条消息可再压一次 —— **允许**，因装载状态已变。  
- **同一**轮用户输入内：上述第 4 点保证最多一次摘要调用。

### 3.6 Token 估算

- **MVP**：采用 **字符数 proxy**（中文按字、英文按词近似）或 **粗算公式** `len/2` 等，留安全系数；后续可换 tiktoken 级库（需评估 Rust 依赖）。
- **目标**：相对比较与阈值触发即可，无需绝对精确。

## 4. 摘要内容与模板护栏（防失真）

### 4.1 必须保留的信息类型

参照 [1201 §7.7](../2026041201/01-Agent会话中Tools状态与大模型感知（设计讨论）.md)：用户目标、约束、已确认事实、未决问题、重要结论；不写编造内容。

### 4.2 工具相关表述

若摘要涉及工具返回：注明「以下为历史工具结果要点，**以当轮系统能力清单为准**」。

### 4.3 与 Skills 一致性（必填句）

摘要头或尾固定短句，例如：

> `[能力状态]` **当前轮可用的工具与权限以对话请求中的系统能力清单及服务端绑定为准；切勿仅根据本摘要判断能否调用某工具。**

避免摘要中出现「用户永远无法使用某 Skill」这类**不可逆**表述；若叙事需要，应写「当时未能调用」等**时态化**描述。

## 5. 存储与数据结构（建议）

### 5.1 最小改动（推荐 MVP）

在 **`conversations`** 表新增列：

- `rolling_summary TEXT` — 当前冷区滚动摘要全文（可为空）。
- `summary_updated_at` — 可选，便于调试与增量策略。
- `compression_pending BOOLEAN NOT NULL DEFAULT 0` — 因 **Skill 装载变更** 而需在 **`chat_send` 内至多消费一次**的压缩请求标记（语义见 §3.5.1）；一次成功处理后清零。

**不入库**的选项：摘要仅存内存 — **不推荐**（进程重启丢失，对话长时行为不一致）。

### 5.2 审计与对比（可选增强）

- 若需「摘要历史版本」：另表 `conversation_summary_snapshots(id, conversation_id, body, created_at)`；MVP 不需要。

### 5.3 与 `messages` 的关系

- **默认**：`messages` **不删改**；仅在构建 API 视图时 **跳过**已纳入摘要范围的旧行（热区外的 user/assistant 不再重复送 verbatim）。
- **特殊**：若采用「摘要也作为一条 `messages`」方案，需 **`role`/`kind` 标记**以免前端误展示 —— 与现有「对话区过滤 system」逻辑统一评估。**建议 MVP**：摘要**不入 messages 表**，仅 `conversations.rolling_summary` + 运行时注入。

## 6. 在 `chat_send` 中的控制流（逻辑草图）

```
收到 user_text、写入 user 消息成功
  → 组装「完整历史」列表（DB 顺序）
  → 读取 compression_pending；与 token 估算一并输入决策（见 §3.5.1：同轮至多一次摘要 LLM）
  → 估算 token/字符
  → 若超过阈值 **或** compression_pending 要求压缩：
        取热区外消息块 → 调摘要 LLM（或分块）→ 更新 rolling_summary；清零 compression_pending
  → 若仅 compression_pending 且历史过短、不调用 LLM：按 §3.5.1 推荐路径清零 pending
  → 构建 api_messages：
        system（现有） + [注入 rolling_summary 块] + 热区消息（及 tool 策略）
  → run_chat_rounds
  → 写 assistant 消息
```

**并发**：同一 `conversation_id` 若未来有多 Tab，需加锁或版本号；**单会话单线程**可保持现状。

## 7. 风险与对策

| 风险 | 对策 |
|------|------|
| 摘要失真 | 固定模板 + 短护栏 + 热区足够大；允许用户未来「重试摘要」 |
| 额外延迟与费用 | 仅超阈值触发；摘要 prompt 极简化；可配置关闭 |
| 与 DB 历史展示不一致 | UI 仍展示完整 `messages`；仅「模型视图」压缩 |
