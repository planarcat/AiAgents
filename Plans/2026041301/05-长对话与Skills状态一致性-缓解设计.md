## 1. 用户问题

- 即使用 Agent Skills 范式开发 Skills，**仍难以抵消巨大历史对话的影响**。典型现象：
  - 第一次**未**加载某 Skill，助手回答「无法回答」；第二次**已**加载该 Skill 后，**仍**沿用旧结论、不调用工具。
  - 第一次**已**加载 Skill 并得到基于工具调用的回复；第二次**卸载**该 Skill 后，**仍**给出仿佛仍有该工具的回答。
- 希望评估下列方案的可行性与组合方式：
  1. **输出约束**：每次回答说明该回答是「基于历史记忆的复述」还是「基于本轮工具调用的结果」。
  2. **Skill 内容更新后的约束**：在 `description`（或等价声明层）中临时插入「工具已更新」类提示，**直到该 Skill 被成功调用过一次**后删除。
  3. **装卸载后的约束**：每次绑定集合发生变化时，在会话中插入提示（如「你的 skills 列表已变化……」）。
  4. **工具身份变更**：对模型暴露的 `tool_name` 增加 **hash / 版本后缀**，当 Skill **内容**或**装卸载状态**变化时更新，使模型在认知上视为「新工具」；或采用等价方案（如仅 system 中的**清单 epoch**）。

---

## 2. 问题解析

### 2.1 根因（为何单靠「范式文档」不够）

| 因素 | 说明 |
|------|------|
| **注意力稀释** | 长上下文中，早期轮次的助手结论权重仍高，易覆盖「当前轮 tools 可用」的信号。 |
| **行为惯性** | 模型在旧轮次中形成了「不调工具 / 假装已调」的模式，后续轮次易延续。 |
| **软约束** | 仅靠 system 中的原则性描述，在 token 很长时**弱于**「最近几条、与状态强绑定」的显式信号。 |

### 2.2 与仓库已有能力的关系

- **2026041201** 已落地方向：会话侧 **Tools 变更提示**（`insert_tools_changed_notices_for_agent` 等），属于本文件 **方案 3** 的**子集或基础**。
- **本设计**补充的是：**方案优先级、与 1/2/4 的叠加关系、以及「改 tool 名 vs epoch」的工程权衡**；**§7** 记录与当前 `chat.rs` 的**执行对照**。

## 3. 解答：各方案可行性（摘要）

| 方案 | 可行性 | 有效性（预期） | 主要成本 / 风险 |
|------|--------|----------------|-----------------|
| **1 输出归因** | **中高**（提示词即可） | **中偏低**（易成套话，不保证真区分） | 回复变长；错误归因反噬信任 |
| **2 更新后 description 脏标记** | **中**（需状态机） | **中高**（直接改模型读到的工具声明） | 「首次调用」判定、并发与重试 |
| **3 装卸载会话提示** | **高**（与现有 notice 链一致） | **高**（近期 token、强绑定当前集合） | 过频需**去重 / 合并** |
| **4 tool name + hash** | **中（工程）** | **理论有帮助**（新名字减弱旧惯性） | 日志与评测需**逻辑 id**；不宜首选污染 schema |
| **4′ 变体：清单 epoch / 仅 system 版本戳** | **高** | **中高** | 不改 tool 名，运维更简单 |

**综合建议（与讨论结论一致）**

1. **优先强化 / 扩展方案 3**（与 1201 变更提示对齐），保证「状态变化」进入**最近**上下文。
2. 痛点若以 **Skill 内容更新**为主，再叠加 **方案 2**。
3. **方案 1** 作轻量补充（短句即可），不作为唯一防线。
4. **方案 4** 优先采用 **epoch / 摘要块**；若仍不足再评估 **tool name 后缀**，并建立逻辑 id 映射。

## 4. 可选后续增强（非本设计结案条件）

- **可见性**：当前 `TOOLS_CHANGED_NOTICE` 为 **`messages.role = system`**；若需用户可见一行，可另增 UI 提示。
- **去重 / 合并**：多次保存可对每会话各插一条 system；若需合并为单条摘要，可在 `insert_tools_changed_notices_for_agent` 侧扩展。
- 讨论稿中的 **方案 2（行级脏标记）**、**方案 4（tool `name` hash）** 未作为本阶段交付项；**§7** 说明以何种实现满足设计意图。
