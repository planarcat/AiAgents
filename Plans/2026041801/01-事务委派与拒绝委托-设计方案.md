## 1. 用户问题

对 **事务委派**（`request_agent_help` / `skill-builtin-delegate`）做产品与交互上的**重新设计**，包括：

1. **说明文案**改为：当你没有足够能力完成事务时，你可以尝试将事务委派给其他助手帮助你完成。
2. **「不接收委托」不再与助手配置里的入站开关强绑定，而是独立为一个新 Skill**：**拒绝委托**。
3. 明确 **委派执行思路**、**界面 UI 文案状态机**，以及 **寻找委托对象** 的多种方案（遍历 / 广播 / Skills 池）与 **拒绝委托** 在接收侧的行为。

---

## 2. 问题解析


| 维度               | 说明                                                                                                                                                                                                         |
| ---------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **与现状关系**        | 当前 `agent-hub` 中委派为 **MVP**：`skill_dispatch::handle_delegate` 校验源端是否装载事务委派、目标是否存在、`accepts_incoming_delegation` 等，返回提示性字符串，**尚无真实子会话与端到端回复**（见 `skill_dispatch.rs` 注释）。本设计在**不否定 MVP**的前提下，定义**目标态**与分期路径。 |
| **入站开关 → Skill** | 「拒绝委托」作为 Skill，意味着：**默认可接收委托**的语义需与「装载本 Skill = 明确拒绝」对齐；需迁移或映射现有 `accepts_incoming_delegation` 列与 UI，避免双源冲突（见 §4.2）。                                                                                       |
| **UI**           | 委派是**长流程**（寻址 → 协商 → 等待 → 成功/失败/重试），需要 **Hub 事件或阶段字段** 驱动前端展示，与现有 `chat_reply_phase` 类似但步骤更多。                                                                                                              |
| **寻址方案**         | 遍历 / 广播 / Skills 池在 **复杂度、实时性、隐私与实现成本** 上差异大，宜 **分阶段**：先实现一种 **MVP 可落地** 的，其余列为扩展。                                                                                                                         |

## 3. Skill 定义（文案与职责）

### 3.1 事务委派（既有 id，改说明）

- **工具名**（保持与现网一致）：`request_agent_help`（或迁移期别名，待工程定）。
- **说明（description，给模型读）**：建议固定为：
  > 当你没有足够能力完成事务时，你可以尝试将事务委派给其他助手帮助你完成。
- **能力边界**：模型在**判断自身无法独立完成**时调用；参数需能表达 **事务摘要 / 意图 / 约束**（具体 JSON 字段在实现阶段与 `skill_dispatch` 对齐）。

### 3.2 拒绝委托（新 Skill）

- **语义**：当某助手**接收到**来自其他助手的委派请求时，若本助手**装载了「拒绝委托」**，则**自动**返回**拒绝类回复**（不进入正常对话推理），或等价行为由 Hub 短路处理。
- **与「事务委派」关系**：二者独立装载；**未装载**「拒绝委托」且满足入站策略时，才进入「是否接受」的判定逻辑。

## 4. 执行思路与 UI 状态机

以下 **UI 文案** 为**用户可见**或**会话侧状态条**建议（可与 `chat_reply_phase` 风格统一）。

### 4.1 发起方（当前助手）

1. 模型判断自身能力不足 → 调用 **事务委派** 工具，Hub 开始 **寻找可委派对象**。
  - **UI**：`正在寻找委托对象...`
2. **分支 A：找到候选且对方接受**（对方根据自身能力与策略判断，见 §5.3）
  - **UI**：`已确定委托对象：{助手名}`  
  - Hub 将指定事务委托给目标，并等待对方处理结果。  
  - **UI**：`已将{事务摘要}委托给{助手名}，正在等待回复...`
  - **a. 成功**：收到对方**成功反馈**  
    - **UI**：`已得到{助手名}的回复`  
    - 回到**当前助手**的常规事务流程（将结果并入上下文，由模型继续回复用户）。
  - **b. 失败或超时**  
    - **UI**：`{助手名}委派失败，正在寻找下一个可委派对象`  
    - 回到 **寻找** 流程（重试列表中的下一个候选），**除非** 已无候选或达到重试上限。
3. **分支 B：未找到可委派对象**（含：全部被拒绝、无候选、或均被「拒绝委托」拒绝）
  - **UI**：`没有找到可委派对象`  
  - 结束委派子流程，进入**下一流程**（由模型/产品决定：向用户说明、或改需求、或结束）。

### 4.2 终止条件

在 **「成功得到一次委派回复」** 或 **「委派失败且不再重试」** 之后，**结束委派子流程**，进入**下一个**用户可见或模型可见步骤（与 §4.1 末句一致）。

## 5. 寻找委托对象：三种方案

### 5.1 方案 1：遍历（顺序询问）

- **流程**：按**固定顺序**（如 `agents` 表创建时间、名称、最近使用）依次向其他助手发起「是否接受」探测；**第一个明确接受**且通过能力检查的作为目标。**失败**则下一个。
- **优点**：实现简单、无广播基础设施、易推理与测试。  
- **缺点**：延迟高；顺序敏感；**无法**一眼看到全局最优。

### 5.2 方案 2：广播

- **流程**：发起方发**委派广播**；所有助手（或 Hub 过滤后的子集）**并行**收到请求，各自判断：**是否因「拒绝委托」直接拒绝**、**是否有能力**、**是否愿意**；**在时限内**向 Hub 反馈**是否愿意接受**。
- **优点**：延迟低、可择优（如先到先得或按负载）。  
- **缺点**：需要 **消息总线 / 事件总线** 或 Hub 内**多路会话调度**；**隐私与噪音**（无关助手也收到）需治理。

### 5.3 方案 3：全局 Skills 池（注册表）

- **概念**：维护一个**对所有助手可见**的 **Skills 池**（逻辑视图即可）：每种 skill（或能力标签）对应 **注册者列表**（助手 id）。  
例：`天气查询 → [助手1]`，`操作文件 → [助手2]`。  
需要委派时：先按**事务所需 skill** 查池，**仅向注册该 skill 的助手**发定向委托请求。
- **优点**：**精准**、可扩展为「能力市场」；与 **1202 SkillRegistry** 方向一致。  
- **缺点**：需 **注册/注销/心跳**（助手装载变更时更新池）；**冷启动**与**一致性**要设计。

### 5.4 推荐分期（建议）


| 阶段       | 建议                                     | 说明                        |
| -------- | -------------------------------------- | ------------------------- |
| **MVP+** | **方案 1 遍历** + 显式「拒绝委托」短路               | 与现有 SQLite 模型兼容，易落地。      |
| **v2**   | **方案 3 池**（最小集：仅 builtin 技能名 + 注册助手列表） | 与「已装载能力」查询、技能 catalog 对齐。 |
| **v3**   | **方案 2 广播**（或混合：池内广播）                  | 在 Hub 具备多助手会话调度后再上。       |


### 5.5 拒绝委托（接收侧行为）

- 当助手 **A** 收到来自 **B** 的委派请求时：  
  - 若 **A** 装载了 **「拒绝委托」** Skill：Hub **不**调用 A 的完整 `chat_send`，**直接**返回**拒绝回复**给委派协议层（或固定 JSON：`{ "accepted": false, "reason": "rejected_by_policy" }`）。  
  - 若未装载：进入 **是否接受** 判定（能力匹配 + 资源 + 用户策略），再决定接受/拒绝。
