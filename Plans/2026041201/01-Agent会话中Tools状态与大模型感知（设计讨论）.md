# Agent 会话中 Tools 状态与大模型感知（设计讨论）

## 1. 用户问题（原话要点）

用户提出工程难题并框定为**设计方案讨论**，案例为：假设存在「查询、总结最近前端技术发展趋势」类 Tool；装载/卸载前后，模型仍可能给出与**当前**工具状态不一致的回复。用户归纳两点：

1. 大模型似乎**无法感知** Tools 的状态变化；
2. 已卸载 Tool 后，模型仍可能给出**仿佛仍有 Tool** 的回复。

并追问：这是 **调用 Tool 的机制问题**，还是 **大模型未遵守诚信**？

用户需要逐项理解：

1. 大模型调用 Tool 的具体逻辑、涉及文件与代码；
2. 用户期望流程（意图 → 判断能力 → 调用 → 返回）与实际的关系；
3. 模型如何「判断自身是否具有某些能力」；
4. 模型如何判断用户意图需要借助某些能力；
5. 模型如何判断能力与需求相关；
6. Agent 的 Tools 状态变化时，模型如何「感知」。

## 2. 问题解析

### 2.1 范围

- 讨论对象：**本仓库** `Apps/agent-hub` 中 DeepSeek Chat Completions + SQLite 会话历史 + `load_tools` / `dispatch_tool` 的实现路径。
- 不在此展开：其他厂商 API 差异、多模态、纯 Agent 框架（LangGraph 等）的通用理论，仅在与本实现对照时提及。

### 2.2 根因类型（结论预览）


| 现象                | 主因归类                                                                                                   |
| ----------------- | ------------------------------------------------------------------------------------------------------ |
| 装载 Tool 后仍说「无法查询」 | **会话历史锚定** + **无状态 API**：旧 assistant 已承诺「无法」，新轮次易复述；非「调度器未注册工具」。                                       |
| 卸载 Tool 后仍像能查     | **历史中的旧回复/旧 tool 结果仍在上下文** + **文本层幻觉**；**执行层**可由 `agent_allows_tool_by_name` 拒绝真实调用，但**拦不住**模型在纯文本中乱说。 |
| 「机制 vs 诚信」        | 主因是 **架构（无状态请求 + 长历史）与指令遵循**，不宜用「诚信」作技术归因；**机制上** tool 列表与执行门禁是符合 OpenAI 式协议的。                         |

## 3. 解答：逐项回答用户的 1–6

### 3.1 大模型调用 Tool 的具体逻辑（本仓库）

**涉及文件**：`Apps/agent-hub/src-tauri/src/hub/chat.rs`（为主）。

**流程概要**：

1. `**chat_send`**：解析 `conversation_id` → `agent_id` → 读取 Agent 的 `system_prompt` → 持久化用户消息 → `**load_tools(pool, agent_id)**`（`skills` JOIN `agent_skills`）→ 将「能力清单 + 通用规则」拼入 system（若当前实现已包含 `build_capabilities_system_suffix`）→ 读取**完整**消息历史 → 组装 `api_messages`（system + 历史）→ `**run_chat_rounds(..., tools)`**。
2. `**run_chat_rounds**`：每轮请求携带**同一批** `tools`；若响应含 `tool_calls`，将 assistant 与 **tool** 消息写入上下文并继续循环；否则返回最终文本。
3. `**dispatch_tool`**：先 `**agent_allows_tool_by_name**`（再次核对 DB 中该 Agent 是否绑定该 skill），未绑定则返回拒绝字符串；通过后再按工具名分支执行。

**要点**：模型**不持久保存**「上一时刻的 Agent 配置」；每轮可见的「能力」= **当次请求**里的 `tools` + **当次 system** + **库中历史消息**。

---

### 3.2 用户期望流程 vs 实际

**期望**：用户提问 → 分析会话与意图 → 理解需求 → 发现需要某能力 → 自检有无能力 → 有则调用 → 返回结果。

**实际（Chat Completions 范式）**：单次（或多轮 tool 循环）**前向推理**。输入为 **system + 历史 + 当前 user** 与可选 `**tools` schema**；输出为 **文本** 或 `**tool_calls`**。没有 API 暴露的独立「自检」阶段；**「判断有没有能力」在行为上等价于**：模型综合 system、tools 列表描述与历史后的生成策略。

因此：**期望流程是合理的产品叙事**，但**不等于**底层逐步骤、可观测的有限状态机。

---

### 3.3 大模型怎么判断「自身是否具有某些能力」？

在实现层面只有：

- **当次请求中的 `tools` 列表**（及每条 `function.description` / `parameters`）；
- **system 中的文字约束**（例如能力清单、通用规则）。

不存在跨请求的「能力寄存器」。模型被训练为在「列表中有某 function」时更倾向发起对应调用；**列表外**则不应调用（但仍可能**幻觉**出名字，故需要 `**dispatch_tool` 校验**）。

---

### 3.4 怎么判断「用户意图需要借助某些能力」？

主要靠 **语义理解** + **工具描述与用户问题的匹配度**（均由训练与提示决定）。例如「最近前端趋势」→ 若存在「检索/摘要」类工具且描述清晰，则更倾向发起 `tool_calls`。这是**概率性**的，不是形式化证明。

---

### 3.5 怎么判断「某能力可解决需求」或「与需求直接相关」？

同样依赖 **自然语言匹配**（用户问题 ↔ `description` / 参数含义）。工程上可通过：**写好 `skills.description`**、**控制工具粒度**、**在 system 中写清调用时机** 来提高一致性；无法保证 100%。

---

### 3.6 Agent 的 Tools 状态变化，大模型怎么「感知」？

**默认不感知「配置变更」本身**。原因：

- 配置变更写入 DB（`agent_skills`），**不会**自动变成一条新消息；
- 下一轮请求里，模型看到的是 **旧 assistant 文本**（可能含「无法查询」或旧工具结果）与 **新的 `tools` + 新的 system 附录** 的**叠加**。

**「感知」的工程含义**应改为：**每一轮请求显式注入**「当前可用工具」与规则（与 `tools` 参数一致），必要时 **新会话**、**摘要** 或 **用户可见的配置变更提示**，以减少历史与当前能力不一致。

---

## 4. 设计结论（回答「机制问题还是诚信」）

- **调用 Tool 的机制**（本仓库）：按 OpenAI 兼容协议工作；**未绑定则拒绝执行**的设计是合理的；问题不在「调度器坏了」。
- **现象 1、2**：主要来自 **无状态 API + 完整历史带来的锚定与复述**，以及 **文本生成与真实 tool 执行不同步**；用「诚信」概括**不利于**定位可改工程点（system、清单、历史管理、UI 提示）。

## 6. 工程方案候选与业界对照（补充落盘）

本节落盘用户提出的两种设想，并给出与主流实践、本仓库 **Agent skills** 的关系（设计层，非实现承诺）。

### 6.1 用户设想 1：每次发送都带上已有 Tools，并默认加约束提示

- **含义**：每一轮可能触发工具调用的请求中，在 API 中传入完整 `tools`（function schema），并在 **system**（或等价位置）加入行为约束，例如「仅可从下列工具列表中调用」「未列出则不得声称已调用」等。
- **结论**：与 **OpenAI 兼容 Chat Completions + tool use** 的主流做法一致。协议为**无状态**；模型要稳定产生合法 `tool_calls`，通常**每一轮**需要工具的补全请求都应携带 **`tools`**；另加 **能力清单 + 通用规则**（本仓库 `build_capabilities_system_suffix` 即此类）是常见增强。
- **定位**：作为**默认基线**保留，而非临时权宜。

### 6.2 用户设想 2：新会话第一次带 Tools，之后仅在更新时插入「可用 Tools 已变化…」

需拆成两部分，避免混为一谈：

| 子设想 | 是否推荐（在「仍需 function calling」前提下） |
|--------|-----------------------------------------------|
| **仅在 skills 绑定变更时**，向会话插入一条 **system 或用户可见** 说明：可用能力已更新、请以当前为准、使用前重新确认 | **推荐**。缓解历史锚定、明确「配置已变」，与第 3.6 节「感知」的工程含义一致。可与 6.1 **叠加**。 |
| **后续 API 请求不再传 `tools`**，仅靠首轮 Tools + 一条提醒 | **通常不可行**。若仍可能在同一会话中发起新的 **function calling**，模型一般**仍需每轮看到工具 schema**；省略 `tools` 与常见实现及协议预期不符。 |

**省 token** 的业界思路多指向：**缓存长 system/工具块**（视厂商与套餐）、**精简 `description`**、**合并低频工具**，而不是「首轮之后不传 `tools`」。

### 6.3 流行组合（摘要）

- 每轮 **`tools` + system 约束/能力清单**（与 6.1 一致）。
- **配置变更时**插入一条说明，或引导 **新会话**（与 6.2 可行部分一致）。
- **UI** 展示当前已装载能力，降低用户与模型认知差。
- **长会话摘要**，减轻旧 assistant 段落对当前轮的影响。
- **MCP 等协议**：解决工具暴露与对接方式；调用侧多数仍须将**当前可用工具**交给模型，而非「首轮一次管终身」。

### 6.4 Agent skills 能否「单独解决」感知问题

在本仓库中，**Agent skills** 体现为：**DB 绑定**（`agent_skills`）+ **`load_tools`** + **`dispatch_tool` / `agent_allows_tool_by_name`**。

| 能力 | 说明 |
|------|------|
| **能解决** | **权威事实**：某 Agent 有权使用哪些工具；执行时拒绝未绑定调用。 |
| **不能单独解决** | 模型侧「每轮是否看到工具定义」「是否被历史带偏」——仍依赖 **每轮传入 `tools` + system**，以及 **变更提示 / UI / 摘要** 等配合。 |

**一句话**：skills 管 **权限与事实**；「感知与行为一致」需 **6.1 +（可选）6.2 可行部分 + 产品层提示** 共同完成。

### 6.5 内置提示词与 Tools 变更提示（产品决议 · 已落地 `Apps/agent-hub`）

- **每轮内置（不对用户展示原文）**：在 `chat_send` 中于 Agent `system_prompt` 之后追加 `BUILTIN_RUNTIME_SYSTEM_SUFFIX`（诚实、仅调用已列出工具、委派或说明无法获取）及 `build_capabilities_system_suffix`（能力清单）；**不写入** `messages` 表。
- **诚实声明**：**不作为 Tool**；`load_tools` 排除 `skill-builtin-honesty`；无全局必选技能（事务委派等为可选）；若模型仍调用 `honesty_acknowledge`，`dispatch_tool` 返回废弃说明。
- **工具边界**：仅可调用已装载工具；若无对应能力且可委派则 `request_agent_help`，否则在回复中说明无法获取相关信息等。
- **Tools 更新**：`agents_update` 在 `agent_skills` 集合相对保存前发生变化时，向该助手下**全部会话**插入 `role=system` 的固定提示；`chat_send` 将 `messages` 中的 `system` 行拼入 API；前端以浅色横幅展示系统提示。

## 7. 长会话摘要策略（方案枚举）

目的：在 **token 预算有限** 时，减少「旧 assistant 段落」对当前轮的锚定，同时**尽量保留**仍会影响后续推理的信息（用户目标、已确认事实、未决事项、工具调用结论等）。以下为常见设计选项，可组合使用。

### 7.1 滑动窗口（只保留最近 K 轮）

- **做法**：仅把最近 N 条 user/assistant（或按 token 截断）送入模型；更早消息丢弃或不再传入。
- **优点**：实现简单、延迟低、无额外模型调用。
- **缺点**：易丢失早期约束、长期目标、已承诺结论；对「长任务」不友好。

### 7.2 周期性摘要（异步或同步）

- **做法**：每当消息数或 token 数超过阈值，用**单独一次**（或廉价小模型）对「将被移出窗口的旧段」生成摘要，再以 **一条 system 或 user 消息**（如「【会话摘要】…」）插入到**历史前端**或**紧接在 system 之后**。
- **变体**：**递归摘要**——新摘要 = compress(旧摘要 + 新移出的段落)，控制摘要长度上限。
- **优点**：保留跨轮上下文；**可专门强调**「当前仍有效的工具能力」「用户最终目标」。
- **缺点**：多一次调用成本；摘要可能失真，需**模板约束**（见 7.7）。

### 7.3 分层记忆（结构化 vs 纯文本）

- **做法**：除文本历史外，维护结构化字段（JSON）：`user_goals`、`decisions`、`open_questions`、`last_tool_results` 等；每轮或压缩时**更新**该结构，再与最近消息一起拼进 prompt。
- **优点**：关键信息不易被摘要冲掉；便于与「能力清单」对齐。
- **缺点**：需要 schema 设计与维护逻辑。

### 7.4 工具与 RAG 相关段落

- **做法**：对含 **tool 消息**的长链：要么保留**最近 M 次** tool 往来的完整原文，要么只保留「工具名 + 参数摘要 + 返回结果摘要」；避免把超长 JSON 原样反复塞进上下文。
- **与**：`agent-hub` 的 `messages` 存库方式结合时，需定义「何时折叠、何时保留」规则。

### 7.5 摘要触发时机

- **按轮数**：每 10 轮压缩一次。
- **按 token**：上下文接近上限前触发。
- **按事件**：用户点击「总结会话」、配置变更后、新开子任务时强制刷新摘要。

### 7.6 与「Tools 感知」的配合

- 摘要或结构化记忆里**显式写入**「截至某时刻，用户已装载的能力列表」或「用户已变更能力，以 system 中本轮清单为准」；**避免**摘要里写死「无法使用某工具」而后续已装载。
- 与第 6.1 节「每轮带 `tools` + 能力清单」**叠加**：摘要负责**叙事与目标**，**当次**仍以 DB + `load_tools` 为权威。

### 7.7 摘要提示词要点（防失真）

- 要求保留：**用户目标、约束、已确认事实、未决问题、重要结论**；**不要**编造未出现的对话内容。
- 若存在工具调用，**标注**「以下为工具返回的摘要，非模型臆测」。
- 长度上限（如固定 500～800 字）与**可刷新**（新摘要覆盖旧摘要）。

### 7.8 成熟架构与开源参考（是否有现成仓库）

**结论（设计层）**：**有成熟「积木」与范式，很少有「一键解决你方全部约束」的单一仓库**。长会话摘要、工具调用、记忆分层在业界多为**可组合模块**；与 **Tauri + SQLite + 自研 `chat_send`** 栈对接时，通常仍需**适配层**（语言、存储形态、工具协议）。

| 类型 | 常见开源 / 生态（示例，非穷尽） | 说明 |
|------|----------------------------------|------|
| **Agent + 记忆 / 摘要** | **LangChain**（Python/JS）：`ConversationSummaryMemory`、`ConversationSummaryBufferMemory`、消息窗口截断等 | 模式成熟、文档多；需自建服务或与后端集成，非直接嵌入 Rust 桌面端。 |
| **状态机式 Agent** | **LangGraph**（Python） | 检查点、多步图、易加「摘要节点」；同上，偏服务端。 |
| **RAG + 会话** | **LlamaIndex**、**Haystack** | 偏检索与 pipeline；记忆/摘要可组合。 |
| **多 Agent** | **AutoGen**、**CrewAI** 等 | 协调与对话历史策略各异，摘要需自配。 |
| **微软生态** | **Semantic Kernel**（.NET / Python 等） | `ChatHistory`、插件与 memory 模式清晰。 |
| **长记忆产品化** | **Mem0**、各类「AI memory」SaaS / 开源 | 强调跨会话持久记忆，与本节「单会话内摘要」有重叠但不等同。 |
| **工具暴露协议** | **MCP**（Model Context Protocol） | 规范工具/资源如何接入客户端；**不替代**「每轮传 tools + 摘要策略」的决策。 |
| **Rust / 本地** | 相对 Python 生态，**一站式「Agent+摘要+Tool」框架较少** | 多在应用内自研（如当前 `agent-hub`），或经 **FFI / 子进程** 调 Python/Node 侧 Agent 库。 |

**落地预期**：可借鉴上述项目的**接口划分**（何时摘要、如何折叠 tool 消息、如何与 checkpoint 同步），但**很少**能不经改造直接替换 `Apps/agent-hub` 的对话存储与 `chat_send` 管线。选型时以「能否迁入你的技术栈」为第一约束。

### 7.9 长会话摘要：综合选型方案（默认推荐）

在**暂不增加打包体积、不引入侧车运行时**（第 8 节方案搁置）的前提下，将第 7.1～7.7 的枚举**组合成一条可落地的默认策略**，便于在 `agent-hub` 内逐步实现。

| 层级 | 选用策略（对应小节） | 默认建议 | 说明 |
|------|----------------------|----------|------|
| **① verbatim 区** | 7.1 滑动窗口 | 始终保留**最近 N 轮** user/assistant **原文**（N 建议 **6～12**，可配）；或按 **token 预算**为「热区」设硬顶，二者取先触发的限制。 | 保证近期指令与语气不被摘要扭曲；实现成本最低。 |
| **② 冷区压缩** | 7.2 周期性 / 递归摘要 | 当「热区之前」仍有过长历史时，对**移出热区**的段落做**一条滚动会话摘要**（可递归：新摘要 = 压缩(旧摘要 + 新移出段)），**单条摘要**设长度上限（如 **600～1000 字** 或可配 token）。 | 避免上下文爆炸；摘要**一条**插入在 system 之后、verbatim 历史**之前**（或与现有 `build_capabilities_system_suffix` 顺序协调）。 |
| **③ 工具链** | 7.4 | **最近 M 次**「含 tool 的多轮片段」保留相对完整（M 建议 **1～3**）；更早的 tool 往来只保留 **工具名 + 参数要点 + 返回要点** 一行式摘要，超长 JSON **截断**。 | 兼顾可调试与省 token。 |
| **④ 轻量结构化（可选）** | 7.3 | **首阶段可不做**全量 JSON 记忆；若做，仅维护极小字段：`user_goal`（一句）、`open_questions`（列表），在**每次生成冷区摘要时**由模型或规则更新，与 ② 同存或附在摘要头。 | 降低「只有摘要丢目标」的风险，而不上复杂 schema。 |
| **⑤ 触发** | 7.5 | **主触发**：估算上下文 token，在达到模型上限的 **~70%～80%** 前执行一次冷区折叠；**辅触发**：用户显式「总结以上」；**事件触发**：Agent skills 变更后，下一轮在摘要或 system 附一句「能力以本轮清单为准」（与第 6 节一致）。 | 避免每轮都摘要。 |
| **⑥ 与 Tools 感知对齐** | 7.6 + 第 6 节 | 摘要正文**禁止**写死「永久无法使用某能力」；写明「**工具与权限以当轮 system 能力清单及 DB 绑定为准**」。每轮请求仍 **`load_tools` + 能力清单后缀 + 完整 `tools`**。 | 摘要管叙事，**权限管事实**。 |
| **⑦ 摘要 LLM 调用** | 7.7 | 单独一次补全，使用固定模板：保留目标/约束/已确认事实/未决；不编造；工具返回注明来源；输出长度上限。 | 与对话模型可用同一 API，模型可选较小/便宜 preset（若日后支持）。 |

**一句话**：**热区滑动窗口 + 冷区单条滚动摘要 + tool 链特殊折叠 + token 阈值触发 + 摘要提示词护栏 + 与每轮能力清单正交**；轻量结构化按需再加。

**与第 8 节关系**：若未来接受增加运行时与体积，可将 ②⑤ 的实现**迁移**到 LangGraph 等，但**表 7.9 的语义约束（⑥⑦）仍然成立**。

## 8. 引入成熟开源：选型路径与「可接受部分重来」时的做法

在**不想重复造轮子**、又**可以接受替换部分模块**的前提下，建议按**决策顺序**推进，避免先选框架再反推需求。

### 8.1 先冻结「不可弃」与「可替换」

- **建议保留（除非你有意整体换桌面技术栈）**：**Tauri 壳 + 前端 UI**、与产品绑定的 **SQLite 业务库**（agents、conversations、skills 绑定等）——可继续作为**权威配置与展示层**。
- **最常被替换的「轮子」**：**纯 Rust 自研**的「多轮补全 + tool 循环 + 记忆策略」——若你愿意引入 **Python/Node 侧 Agent 运行时**，这一块可整块外包给 **LangGraph / LangChain** 等；**或**保留 Rust 调 LLM，只把「摘要 / 记忆」交给独立服务。
- **边界要画清**：密钥、用户数据落盘、是否本地优先——影响「嵌入式 Python」vs「本机独立进程」vs「远端 SaaS」。

### 8.2 三条常见集成形态（由浅入深）

| 形态 | 做法 | 适用 |
|------|------|------|
| **A. 侧车服务（Sidecar）** | Tauri/Rust 仍管 UI 与 DB；**单独进程**跑 Python（FastAPI）或 Node，实现 LangChain/LangGraph 对话管线；Rust 通过 **HTTP/WebSocket/IPC** 发「用户一句 + conversation_id」，对方返回 assistant 文本及可选 tool 审计信息。 | 想少动 SQLite 与 UI；接受多进程与打包体积（Python 运行时）。 |
| **B. 仅替换「大脑」管线** | Rust **只保留**：`load_tools`、**`dispatch_tool`（真实执行仍在你侧）**；把「拼 message、摘要、多轮 tool」迁到 LangGraph，但 **tool 执行回调**仍调回 Tauri command 或本地 HTTP（**工具执行不外包给不可信远端**）。 | 重视记忆/摘要成熟度，且要坚持本地执行工具。 |
| **C. 渐进绞杀（Strangler）** | 新会话走新管线，旧会话仍走旧 `chat_send`；或开关切换 A/B。 | 降低一次性推倒风险。 |

### 8.3 选型检查清单（落地前必答）

1. **运行时**：能否接受安装/打包 **Python 3.x** 或 **Node**？若不能，可选范围缩到 **Rust 库 + 自研** 或 **远端托管 Agent**（合规与隐私另议）。
2. **状态存哪**：对话与摘要存在 **对方框架的 checkpoint**（如 LangGraph SQLite/Postgres）还是仍用你方 `messages` 表？若双写，需定义**单一真相源**。
3. **工具协议**：对方框架的 tool 定义能否 **映射** 到你现有 `skills` 表 + `dispatch_tool`？通常要写一层 **adapter**（名称、参数 JSON、执行回调）。
4. **摘要与感知**：沿用第 6～7 节结论——**每轮仍须**让模型看到当前 `tools`；开源框架负责「何时压缩历史」，**能力权威**仍以你库为准。

### 8.4 推荐推进顺序（设计阶段）

1. **Spike（1～3 天）**：用候选框架做一个**最小**「多轮 + 一假工具」示例，验证 **Tauri → 侧车** 延迟与打包可行性。  
2. **接口契约**：冻结 `POST /chat { conversation_id, user_text }` 与返回结构；tool 执行是否同步回调 Rust。  
3. **再决定推倒范围**：只换「对话编排」还是连「消息表结构」一起换。

### 8.5 风险简述

- **依赖与体积**：Python 生态功能多，打包与升级链更重。  
- **双栈维护**：Rust + Python 团队认知成本。  
- **版本漂移**：框架升级可能改 memory API，需锁版本与回归测试。
